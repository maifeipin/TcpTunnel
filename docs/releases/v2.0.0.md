# v2.0.0

## Summary
Major upgrade for the Lite tunnel workflow. This version introduces multi-proxy TCP forwarding with JSON configuration while keeping one-click deployment and backward compatibility for legacy single-port mode.

## Highlights
- Added JSON-based client config with multiple proxy rules.
- Added dynamic public port listeners on server side based on client registration.
- Added support for common scenarios in one client process:
  - RDP to Windows desktop
  - SSH to Linux host
  - HTTP and HTTPS access to LAN web consoles
- Kept legacy compatibility:
  - Old single-rule client arguments still work
  - Legacy server single public port mode still available
- Updated installer and packaging scripts to keep operation simple.

## What Changed
- Client now supports tunnel-client.json with multiple proxies.
- Client control channel now sends proxy registration to server.
- Server now routes sessions by proxy name and session id.
- Server can open multiple public ports according to active proxy rules.
- Build and install scripts updated for new configuration flow.
- Documentation updated with multi-scenario examples.

## Typical Scenarios
- RDP: mstsc /v:VPS_IP:33890
- SSH: ssh -p 2222 user@VPS_IP
- HTTPS panel: https://VPS_IP:8443

## Breaking Notes
- No hard breaking changes for existing users in compatibility mode.
- Recommended migration path is to use tunnel-client.json for multi-rule setup.

## Upgrade Steps
1. Deploy latest server binary and restart service.
2. Ensure cloud security group allows control, data, and required public ports.
3. Regenerate client package with the updated build script.
4. Edit tunnel-client.json to add all needed proxy rules.
5. Start client and verify each mapped service.

## Security Notes
- Use strong AUTH_KEY in production.
- Minimize exposed public ports to only required services.
- Prefer source-IP restrictions on control/data ports.
